# ecommerce-platform 当前项目运行说明

本文说明仓库当前可实际运行的本地开发路径：**Docker Compose 只启动基础依赖，五个核心 Java 服务运行在宿主机**。这不是全量容器化部署，也不依赖云账号或外部密钥。

## 运行范围

本地启动的核心服务如下：

| 服务 | 宿主机端口 | Health endpoint |
| --- | ---: | --- |
| Catalog Service | 8080 | `http://localhost:8080/actuator/health` |
| Search Service | 8081 | `http://localhost:8081/actuator/health` |
| Inventory Service | 8082 | `http://localhost:8082/actuator/health` |
| Order Service | 8083 | `http://localhost:8083/actuator/health` |
| Payment Service | 8084 | `http://localhost:8084/actuator/health` |

Compose 启动的依赖为 Spanner Emulator、Kafka、Zookeeper、Redis 和 Elasticsearch。它们并不包含 Java 服务。

当前仓库没有可用的 Gateway、User、Cart 或 Notification 本地启动流程；也没有 Kubernetes、GCP 部署目录或 CI 配置。这些内容仅为规划，不应视作已实现功能。

## 前置条件

- Docker Desktop（或可用的 Docker daemon）和 Docker Compose v2；Windows 请先启动 Docker Desktop。
- JDK 21。
- Maven（`mvn` / Windows 下的 `mvn.cmd`）并已加入 `PATH`。
- Linux/macOS：Bash、`curl`。
- Windows：PowerShell；首次启动 Kafka topic 初始化还需要 Git for Windows 的 Git Bash。脚本会优先使用 `C:\Program Files\Git\bin\bash.exe`，不会依赖未配置的 WSL。
- 可选前端：Node.js LTS 与 npm。

不需要 Google Cloud 凭据。`local` profile 使用 Spanner Emulator。

## 一键启动后端与基础依赖

在仓库根目录执行以下命令。**该命令不会启动 React 前端**；它只会启动 Docker 基础依赖和五个核心 Java 后端服务。

### Windows PowerShell

```powershell
.\scripts\run-local.ps1
```

### Linux / macOS

```bash
./scripts/run-local.sh
```

脚本将执行以下操作：

1. 启动 Compose 依赖并等待端口就绪。
2. 初始化 Spanner Emulator 的 instance、database 和 DDL。
3. 创建代码实际使用的 Kafka topics。
4. 执行 `mvn -q -DskipTests install`。
5. 使用 `local` profile 在宿主机依次启动 Catalog、Search、Inventory、Order、Payment。
6. 等待每个服务的 `/actuator/health` 返回成功。

启动过程的 Java 服务日志位于 `.local/logs/`，进程 ID 位于 `.local/pids/`。

## 初始化本地演示商品

后端启动成功后，在仓库根目录运行演示数据脚本。该步骤会通过真实的 Catalog local API 和 Inventory local seed API 创建或更新 20 个固定 SKU，并写入库存、同步 Redis、预热抢购价格。

### Windows PowerShell

```powershell
.\scripts\seed-demo-data.ps1
```

### Linux / macOS

```bash
./scripts/seed-demo-data.sh
```

脚本可重复执行：已存在的 Catalog SKU 使用真实 PATCH 接口更新，不存在时才创建；Inventory 使用真实 seed 接口恢复为清单中的演示库存。固定 SKU ID 包括 `demo-phone-001`、`demo-headphones-001`、`demo-keyboard-001`、`demo-monitor-001` 等。商品定义统一保存在 `scripts/demo-products.tsv`。

## 启动前端（单独终端）

后端健康检查通过后，另开一个终端执行以下命令。前端不是 `run-local` 脚本的一部分。

### Windows PowerShell

```powershell
cd frontend
npm install                # 首次安装或 package-lock.json 变化后执行
Copy-Item .env.example .env # 首次执行
npm run dev
```

### Linux / macOS

```bash
cd frontend
npm install                # 首次安装或 package-lock.json 变化后执行
cp .env.example .env       # 首次执行
npm run dev
```

`.env.example` 已预置上述 20 个固定 demo SKU ID，不需要手动编辑。Vite 默认输出访问地址（通常为 `http://localhost:5173`）。它将 `/catalog-api`、`/inventory-api`、`/order-api`、`/payment-api` 代理到已经在宿主机运行的后端服务；页面商品卡片的名称和价格来自真实 Catalog batch 查询，可用库存来自 Inventory 的 local-only 查询接口。库存是否足够仍以抢购请求时 Inventory 的原子预留结果为准。

## 分步启动

当只想重新启动 Java 服务，或需要在启动前排查依赖时，可分两步进行。

### Windows PowerShell

```powershell
.\scripts\run-local.ps1 -Mode deps-only
.\scripts\run-local.ps1 -Mode services-only
```

### Linux / macOS

```bash
./scripts/run-local.sh --deps-only
./scripts/run-local.sh --services-only
```

`deps-only` 会启动和初始化 Docker 依赖；`services-only` 假定依赖已就绪，只启动宿主机 Java 服务。

## 容器地址与宿主机地址

必须区分 Docker 容器内部网络和宿主机网络：

| 依赖 | 容器内部地址 | 宿主机 Java 服务使用的地址 |
| --- | --- | --- |
| Spanner Emulator gRPC | `spanner:9010` | `localhost:9010` |
| Kafka | `kafka:9092` | `localhost:29092` |
| Redis | `redis:6379` | `localhost:6379` |
| Elasticsearch | `elasticsearch:9200` | `http://localhost:9200` |

`infra/local/.env` 仅供 Compose 使用，因此其中的 `spanner:9010` 与 `kafka:9092` 不能直接用于宿主机启动的 Spring Boot 服务。宿主机变量样例见 [host.env.example](../infra/local/host.env.example)。启动脚本已自动设置这些变量。

## 为什么必须使用 local profile

Catalog 的本地 SKU 管理接口和 Inventory 的本地库存 seed 接口仅在 `local` profile 可用。启动脚本会设置：

```text
SPRING_PROFILES_ACTIVE=local
SPANNER_EMULATOR_HOST=localhost:9010
KAFKA_BOOTSTRAP_SERVERS=localhost:29092
```

同时会排除需要 Application Default Credentials 的 Spring Cloud GCP Spanner 自动配置，使服务可直接使用本地 Emulator。

## 最小交易 smoke test

核心服务全部健康后，运行：

### Windows PowerShell

```powershell
.\scripts\smoke-local.ps1
```

### Linux / macOS

```bash
./scripts/smoke-local.sh
```

脚本会：

1. 检查五个核心服务的 health endpoint。
2. 通过 Catalog local-only API 创建 SKU。
3. 通过 Inventory local-only API 写入库存。
4. 调用真实的库存预留 API，并取得其返回的 `orderId`。
5. 通过 `GET /api/v1/orders/{orderId}` 等待真实订单状态达到 `PENDING_PAYMENT`。
6. 调用 Payment 的支付成功接口。
7. 再通过 Order HTTP API 等待订单状态变为 `PAID`。

这验证的是实际的 Kafka/Outbox 异步链路；Payment 的 HTTP 成功响应本身不等同于 Order 的 `PAID`。

## 订单查询 API

Order Service 提供最小只读查询接口：

```text
GET http://localhost:8083/api/v1/orders/{orderId}
```

成功时返回：

```json
{
  "orderId": "o-fs-A100-01-...",
  "skuId": "sku-1",
  "userId": "local-user",
  "quantity": 1,
  "status": "PENDING_PAYMENT",
  "createdAt": "2026-07-15T20:00:00Z",
  "updatedAt": "2026-07-15T20:00:00Z"
}
```

不存在的订单返回 HTTP 404。该接口不会返回 Outbox、幂等键或其他内部实现字段。

## 前端行为与质量检查

Vite 会将以下路径代理至宿主机服务：

- `/catalog-api` → `http://localhost:8080`
- `/inventory-api` → `http://localhost:8082`
- `/order-api` → `http://localhost:8083`
- `/payment-api` → `http://localhost:8084`

前端从 Inventory 预留响应取得真实 `orderId`，每两秒查询一次 Order API。`PAID`、`CANCELLED`、`FAILED` 为终态，达到终态后停止轮询；轮询超过 60 秒也会停止并提示。Payment 成功响应不会直接让页面显示 `PAID`。

前端质量检查：

```bash
npm run lint
npm run typecheck
npm test
npm run build
```

## 停止与清理

停止 Java 服务和 Compose 依赖：

```powershell
.\scripts\stop-local.ps1
```

```bash
./scripts/stop-local.sh
```

以上命令会停止 Java 服务并执行 `docker compose down`。前端开发服务器需要在它自己的终端使用 `Ctrl+C` 停止。

### 每次重启后数据是否保留？

不是所有数据的行为都相同：

| 操作 | Spanner 订单/SKU/库存数据 | Redis | Kafka topics / 消息 | Elasticsearch 索引 |
| --- | --- | --- | --- | --- |
| 仅再次执行 `run-local`，且容器仍在运行 | 保留 | 保留 | 保留 | 保留 |
| 执行 `stop-local` 后再执行 `run-local` | **不保留** | **不保留** | 保留 | 保留 |
| 执行 `infra/local/scripts/reset.sh` | **不保留** | **不保留** | **不保留** | **不保留** |

原因是当前 Compose 没有为 Spanner Emulator 和 Redis 配置数据卷：`docker compose down` 会删除它们的容器，因此两者的数据会消失。Kafka、Zookeeper、Elasticsearch（以及 Grafana）使用命名 Docker volume，普通 `down` 不会删除这些数据卷，所以 Kafka topic、消息和 ES 索引会保留。

因此，通常建议：

- 只重启 Java 服务时，保持 Compose 容器运行，并使用 `services-only` 模式；数据会保留。
- 执行 `stop-local` 后，下一次启动应视为新的 Spanner/Redis 本地环境；可以重新运行 smoke test 创建测试 SKU 和库存。
- 需要把 Kafka、Elasticsearch 也清空时，执行 reset。

需要彻底删除所有使用命名卷的本地依赖数据并重新初始化时，使用：

```bash
./infra/local/scripts/reset.sh
```

该操作会删除本地 Compose 数据卷，下一次运行需要重新创建 Spanner schema 和 Kafka topics。

## 常见问题

### Docker daemon 不可用

出现 Docker named pipe 或 daemon 连接错误时，启动 Docker Desktop，等待其状态为 Running 后重新执行启动脚本。

### Kafka topic 初始化失败（Windows）

确认已安装 Git for Windows，且 `C:\Program Files\Git\bin\bash.exe` 存在。不要依赖没有 Linux 发行版的 WSL `bash.exe`。

### 服务健康检查超时

检查 `.local/logs/` 中对应服务的最新 `.log` 与 `.log.err`。常见原因包括 Docker 依赖尚未就绪、9010/29092 等端口被占用，或 Maven/JDK 版本不符合要求。

### 前端无法加载商品

Catalog 当前没有通用商品列表 API，前端调用真实的 batch SKU 查询接口。请确认已经运行 `seed-demo-data`，并从最新的 `frontend/.env.example` 复制生成 `frontend/.env`；旧 `.env` 不会因仓库中的 example 更新而自动变化。
